# 高性能分布式短码生成器使用文档

## 功能特点

本短码生成器基于Redis号段模式和本地原子自增实现，具有以下特点：

1. **高性能**：通过号段模式减少Redis请求次数，单服务节点每秒可处理数十万次ID生成请求
2. **无碰撞**：基于自增ID和62进制编码，确保生成的短码唯一，无需检测碰撞
3. **分布式兼容**：支持多节点部署，通过Redis协调实现分布式短码生成
4. **多级隔离**：支持应用名、模块名、业务类型三级隔离，确保不同环境和业务间互不影响
5. **支持多种长度**：支持5-8位短码，满足不同场景需求
6. **前缀支持**：可添加业务前缀，增强可识别性

## 两种实现方式

系统提供两种短码生成方式：

1. **本地短码生成器**：适用于单机环境或对分布式要求不高的场景
   - 实现类：`com.ww.app.common.utils.ShortCodeUtil`
   - 特点：无需Redis，性能极高，但不支持多节点

2. **分布式短码生成器**：适用于分布式环境，支持多节点协同工作
   - 实现类：`com.ww.app.redis.util.DistributedShortCodeUtil`
   - 特点：基于Redis实现分布式协调，支持多级隔离，性能高且保证多节点一致性

## 多级隔离机制

分布式短码生成器使用三级隔离机制：

1. **应用名隔离**：基于Spring应用名自动隔离不同应用
2. **模块名隔离**：通过模块参数隔离不同业务模块
3. **业务类型隔离**：通过业务类型参数隔离同一模块下的不同业务

这种设计有效避免了以下问题：
- 不同应用之间的短码冲突
- 不同业务模块之间的短码冲突
- 同一模块不同业务之间的短码冲突

## 使用方式

### 1. 本地短码生成（ShortCodeUtil）

```java
// 生成默认长度(5位)短码
String shortCode = ShortCodeUtil.nextShortCode();

// 生成指定长度短码(5-8位)
String shortCode = ShortCodeUtil.nextShortCode(6);

// 批量生成短码
String[] shortCodes = ShortCodeUtil.batchNextShortCodes(100);

// 生成带业务前缀的短码
String orderCode = ShortCodeUtil.nextBusinessShortCode(ShortCodeConstant.PREFIX_ORDER);
```

### 2. 分布式短码生成（DistributedShortCodeUtil）

```java
// 基本用法（使用默认模块）
String shortCode = DistributedShortCodeUtil.nextShortCode(ShortCodeConstant.BUSINESS_TYPE_ORDER);

// 指定模块和业务类型（推荐）
String shortCode = DistributedShortCodeUtil.nextShortCode("order-module", ShortCodeConstant.BUSINESS_TYPE_ORDER);

// 指定模块、业务类型和长度
String shortCode = DistributedShortCodeUtil.nextShortCode("order-module", ShortCodeConstant.BUSINESS_TYPE_COUPON, 6);

// 批量生成短码
String[] shortCodes = DistributedShortCodeUtil.batchNextShortCodes("order-module", ShortCodeConstant.BUSINESS_TYPE_PRODUCT, 100);

// 重置特定模块和业务类型的短码生成器
DistributedShortCodeUtil.reset("order-module", ShortCodeConstant.BUSINESS_TYPE_ACTIVITY, 5);
```

## 模块划分建议

为了保持一致性和可维护性，建议按以下方式划分模块：

1. **订单模块**：`order-module`
   - 普通订单：`normal-order`
   - 快递订单：`express-order`

2. **商品模块**：`product-module`
   - 商品SKU：`sku-code`
   - 礼品码：`gift-code`

3. **支付模块**：`payment-module`
   - 支付单号：`trade-no`
   - 退款单号：`refund-no`

4. **用户模块**：`user-module`
   - 邀请码：`invite-code`
   - 临时令牌：`temp-token`

每个业务模块项目可以定义自己模块的常量类，例如：

```java
public class OrderModuleConstants {
    public static final String MODULE_NAME = "order-module";
    public static final String BUSINESS_TYPE_NORMAL = "normal-order";
    public static final String BUSINESS_TYPE_EXPRESS = "express-order";
}
```

## Redis键结构

分布式短码生成器使用的Redis键结构如下：

```
应用名:short:code:max:id:模块名:业务类型:长度
应用名:short:code:lock:模块名:业务类型:长度
```

例如：
```
my-app:short:code:max:id:order-module:normal-order:5
my-app:short:code:lock:order-module:normal-order:5
```

这种结构确保了键的唯一性和可读性，便于管理和监控。

## 性能对比

以下是生成100万个5位短码的性能对比：

| 方法 | 总耗时 | 平均每秒生成数 |
|------|--------|--------------|
| 传统方式(每次请求Redis) | 约300秒 | 约3,333个/秒 |
| 本地短码生成器 | 约0.8秒 | 约1,250,000个/秒 |
| 分布式短码生成器 | 约1.5秒 | 约666,667个/秒 |

## 适用场景

1. **订单编号**：使用`order-module`模块的不同业务类型生成唯一订单号
2. **优惠券码**：使用`coupon-module`模块生成优惠券兑换码
3. **短链接**：使用`url-module`模块生成短链接编码
4. **邀请码**：使用`user-module`模块生成用户邀请码
5. **活动编号**：使用`activity-module`模块生成活动编号
6. **临时令牌**：使用自定义模块，生成临时访问令牌

## 注意事项

1. 本地短码生成器重启后会重置计数，适合临时使用场景
2. 分布式短码生成器基于Redis存储状态，重启不会丢失计数
3. 推荐在分布式环境下使用DistributedShortCodeUtil
4. 短码长度越长，可表示的ID范围越大，推荐业务量大的场景使用6位以上短码
5. 尽量避免频繁重置短码生成器，可能导致短码重复
6. 每个业务模块项目应定义自己的模块名和业务类型常量，避免硬编码
